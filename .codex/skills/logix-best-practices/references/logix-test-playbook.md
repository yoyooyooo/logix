# Logix Test 手册

## 1) 设计目标

- 测试模型与生产运行模型一致（Pure Effect + Layer 友好）。
- 场景测试具备声明性，运行结果具备可断言与可解释性。
- 支持从单场景验证扩展到 round-trip 与性质测试。

## 2) 基础构件

- 基础 runner：用于直接运行 Effect 测试程序。
- 场景 DSL：先声明主模块/协作模块/依赖层，再执行测试程序。
- 执行结果模型：统一收集状态、动作、错误与诊断，供断言工具消费。

## 3) 场景测试范式

1. 用声明式配置构造测试场景（主模块 + 可选协作模块 + Layer）。
2. 在场景 API 上执行测试程序（dispatch / assert.state / assert.signal）。
3. 用结果断言工具验证关键行为（动作轨迹、状态变化、错误通道）。

## 4) 与测试框架协作

- 简易模式：普通测试框架中运行一次性程序，适合快速接入。
- Effect 原生模式：直接使用 effect 风格测试入口，减少手工桥接代码。
- 原则：测试控制权在测试框架，语义控制权在 Effect/Logix 程序。

## 5) Round-trip 与性质测试

- Identity：Intent/模型往返保持语义一致。
- Stability：代码往返不产生无意义抖动。
- Resilience：面对软/硬变更，系统按约定降级而非崩溃。

可用方法：

- 生成器：随机生成合法输入模型。
- 变异器：结构化修改代码或配置。
- 检验器：验证不变量（结构不塌、语义可追踪、降级可解释）。

## 6) 交付清单

- 每个新增语义至少有一个场景级测试锚点。
- 并发与事务边界都有失败路径断言（不只测 happy path）。
- Round-trip 测试可按周期运行，结果可复现实验参数。
