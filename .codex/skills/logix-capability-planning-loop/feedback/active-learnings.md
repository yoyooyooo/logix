# Active Skill Learnings

## Current Lessons

### Stable Judge Before Self Evolution

Skill feedback is useful only when judged against a stable evaluator. Keep the immutable judge core separate from normal workflow edits.

### Sub-agents Submit Feedback, Main Agent Promotes

Sub-agents may discover reusable boundaries or prompt gaps, but they must return Skill Feedback Packets. The main agent applies promotion gates and writes patches.

### Entering The Skill Activates Multi-Agent Planning

Once the user enters `logix-capability-planning-loop`, the default planning topology is multi-agent. The main agent stays on coordination and control-plane writeback. Planning argument drafting and review pressure should run in sub-agent lanes whenever the platform can execute them.

### Memory Is Guidance, Not Authority

Feedback ledger and active learnings help future agents, but they do not override Coverage Kernel, Decision Policy, Fitness hard gates, or templates.

### Surface Registry Before Snapshot Prose

Shape snapshots are for human reading. Any public concept candidate must first be recorded in `docs/next/logix-api-planning/surface-candidate-registry.md` with owner projection, covered caps, proof state, authority target, and status.

### Probe Local Until Promoted

Implementation probes can supply executable pressure, but probe-local success does not freeze API shape or authority law. A probe must end with discard, keep-probe-local, principle promotion, authority request, or implementation-task conversion.

### Global Closure Is Separate From Local Readiness

Proposal readiness and proof success are local signals. Total API shape readiness requires the Global API Shape Closure Gate across scenarios, caps, projections, collisions, proofs, surface candidates, authority writebacks, snapshot sync, and housekeeping.

### Probe Code Needs Lifecycle

Verification probes may add temporary code to grow tests and expose boundaries. Before reuse or wave closure, record whether each probe artifact stays probe-local, becomes retained harness, gets generalized into internal substrate, moves to test fixture, gets deleted, or becomes an authority request.

### Scenario Names Need Naming Review

Scenario, witness, or packet-oriented names are acceptable while proving a narrow VOB. Reuse across proof lanes must trigger naming review so temporary local vocabulary does not become final implementation vocabulary by accident.

### Paused Deferred Authority Intake Needs Explicit Approval

When run-state is paused and only deferred authority-intake remains, generic continue is not productization approval. Keep the loop paused or route to housekeeping, stale snapshot refresh, or skill feedback until the user explicitly asks for that authority intake.

### Pressure Must Challenge The Frozen Shape

High-risk pressure is not status quo regression testing. A pressure packet must bundle interacting capabilities, force plausible counter-shapes, make the no-change path carry proof burden, and use implementation friction probes when document reasoning cannot decide.

### Taste Needs Its Own Discipline

Human first-read, adoption friction, and API taste are valid pressure lanes after capability coverage appears closed. They must use `references/taste.md`, split Agent cost from human cost, run overfit / underfit guard, carry human status quo burden for no-change decisions, and close with a decision latch. Taste can challenge shape quality, but it must not bypass P0 laws, concept-count, public-surface containment, Agent-first stability, proof obligations, or concept admission.

### Final Assets Must Read Cleanly

After a probe or pressure wave is consumed, production code and formal test file names should read as final behavior, boundary, contract, or guard assets. Keep `CAP-*`, `PF-*`, `SC-*`, pressure packet, and task ids in comments, metadata, or docs links for traceability. Process-oriented names may remain only in quarantined fixture, support, or harness paths with lifecycle and cleanup trigger.
