Logix 在 `@logixjs/core` 与 `@logixjs/react` 之间提供的组合点并不多，业务开发里通常只需要先回答几类稳定的问题：

- 我应该用 **单例** 还是 **多实例**？
- 子模块应该跟随谁的生命周期？UI 应该怎么拿到“属于这个父实例”的子模块？
- 什么时候需要 `RuntimeProvider.layer` 这样的“局部差异化配置”？

## 一句话原则（默认路径优先）

1. **单例/多实例要显式表达**：`useModule(ModuleTag)` 是 Provider 环境里的单例；`useModule(Program, { key })` 才是多实例。
2. **父子组合优先用 imports（strict scope）**：子模块跟随父实例；Logic 用 `$.imports.get(Child.tag)`，UI 用 `host.imports.get(Child.tag)`。
3. **跨模块协作优先放在 owner logic 里**：当协作属于页面或父模块时，把它留在 `Logic`，通过 imports scope 解析子模块。
4. **UI 默认只绑定 Host，一跳即可**：深层组件不要到处“爬 imports 树”；在边界 resolve 一次，把 `ModuleRef` 往下传。

## 常见误用（直接避开）

1. **把 imports 子模块当成全局单例去用**：在 UI 写 `useModule(Child.tag)`，但你其实要的是 `host.imports.get(Child.tag)`。
2. **用 root 读取去“挑一个实例”**：多实例协作优先由拥有者模块在其 scope 内 `$.imports.get(...)` 驱动，不要把 root 读取当作实例选择器。
3. **深层组件到处爬树**：`host.imports.get(A).imports.get(B)...` 让依赖关系分散且难调试；在边界 resolve 一次往下传。
4. **希望局部覆盖影响 Provider 单例**：`useModule(ModuleTag)` 读取的是当前 `RuntimeProvider` scope；需要可覆盖语义时，改用 imports 或 `RuntimeProvider.layer`。
