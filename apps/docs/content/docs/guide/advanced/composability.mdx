Logix exposes a small set of composition points across `@logixjs/core` and `@logixjs/react`. Most application authoring decisions reduce to a few repeatable questions:

- Should this be a **singleton** or **multi-instance**?
- If a module composes child modules, how does the UI get the child **within the parent instance scope**?
- When do you need subtree overrides like `RuntimeProvider.layer`?

## One-sentence rules (prefer the default path)

1. **Make singleton vs multi-instance explicit**: `useModule(ModuleTag)` is a Provider-scoped singleton; `useModule(Program, { key })` is multi-instance.
2. **For parent-child composition, prefer imports (strict scope)**: children follow the parent instance; Logic uses `$.imports.get(Child.tag)`, UI uses `host.imports.get(Child.tag)`.
3. **Cross-module coordination should stay with the owner logic**: when collaboration belongs to a page or parent module, keep it in `Logic` and resolve children via imports.
4. **UI should bind to a Host by default**: don’t “climb the imports tree” everywhere; resolve once at the boundary and pass `ModuleRef` down.

## Composition cheat sheet (what should I use?)

| What you’re solving | Recommended entry | Scope intuition | Where it lives | Read more |
| --- | --- | --- | --- | --- |
| Local UI state (replace `useState/useReducer`) | `useLocalModule` | component lifetime | React component | [React integration](../recipes/react-integration) |
| Page/session multi-instance (tabs/sessions) | `useModule(Program, { key })` | one instance per `(Program, key)` | React component | [React integration](../recipes/react-integration) |
| Parent-child composition (children follow parent) | `imports` + `$.imports.get(Child.tag)` | strict imports scope | Program-mounted Logic | [Cross-module communication](../learn/cross-module-communication) |
| UI reads child within parent instance scope | `host.imports.get(Child.tag)` / `useImportedModule` | parent instance scope | React component | [useImportedModule](../../api/react/use-imported-module), [Route-scope modals](../recipes/route-scope-modals) |
| Runtime-scope service read | `$.use(Tag)` | current runtime scope | Logic | [Cross-module communication](../learn/cross-module-communication) |
| Subtree env override (light config) | `RuntimeProvider layer={...}` | subtree env override | React tree | [Runtime](../../api/core/runtime) |
| Caller DX (author-side) | handle extensions | extra handle fields | module author code | [Module handle extensions](./module-handle-extensions) |
| Reuse “logic snippets / flow templates” | Pattern functions | `(config) => Effect` or `($, config) => ...` | `guide/patterns` / helpers | [Pattern example](../patterns/optimistic-update) |

## Common footguns (avoid these)

1. **Treating an imports child as a global singleton**: writing `useModule(Child.tag)` when you actually need `host.imports.get(Child.tag)`.
2. **Using root reads to “pick an instance”**: multi-instance collaboration should stay in the owner module or parent page logic via `$.imports.get(...)` and imports scope.
3. **Climbing the imports tree everywhere**: `host.imports.get(A).imports.get(B)...` scatters dependencies; resolve once at the boundary and pass `ModuleRef` down.
4. **Expecting local overrides to affect provider singletons**: `useModule(ModuleTag)` reads from the current `RuntimeProvider` scope; if you need override semantics, use imports or `RuntimeProvider.layer`.
