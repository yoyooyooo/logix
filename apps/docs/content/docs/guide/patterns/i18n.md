---
title: Internationalization (i18n)
description: Recommended patterns for wiring an external i18n instance into Logix (injection, tokens, language switching, async readiness).
---

Logix recommends treating the “translation engine” as an external capability injected into the Runtime, instead of creating a separate instance inside each module. This ensures:

- A single i18n instance is shared within one Runtime tree.
- Logic can read i18n as a normal injected service.
- UI can keep using your existing i18n solution (e.g. i18next + react-i18next).

The examples below use the minimal adapter provided by `@logixjs/i18n`.

## 1) Inject an external i18n instance

`@logixjs/i18n` defines a minimal `I18nDriver` shape and provides `I18n.layer(driver)` for injection:

```ts
import * as Logix from "@logixjs/core"
import { Layer } from "effect"
import { I18n } from "@logixjs/i18n"

export const AppRuntime = Logix.Runtime.make(RootProgram, {
  layer: Layer.mergeAll(
    I18n.layer(i18nDriver),
    AppInfraLayer,
  ),
})
```

If you use `RuntimeProvider.layer` in React, you can also put `I18n.layer(i18nDriver)` at the **outermost** Provider (the one that provides `runtime={...}`), so the whole subtree shares the same default i18n service.

## 2) Get i18n in Logic (service-first default)

Inside module Logic, use `$.use(I18nTag)` to read the injected i18n service:

```ts
import { Effect } from "effect"
import { I18nTag } from "@logixjs/i18n"

export const MyLogic = MyModule.logic(($) =>
  Effect.gen(function* () {
    const i18n = yield* $.use(I18nTag)
    // ...
  }),
)
```

## 3) Message tokens (recommended: put “what to translate” into state)

If your goal is replayable/auditable state, a good default is to generate a **message token** in Logic and write it into module state:

```ts
import { token } from "@logixjs/i18n"

const msg = token("form.required", { field: "name" })
```

In UI, translate the token into a final string (e.g. `t(msg.key, msg.params)`). After language switches, the token itself stays the same, while the rendered text updates with the i18n instance.

> Token `params` should be serializable (`null/boolean/number/string`, etc.). Don’t put big objects in it, and don’t pass fields like `lng/lngs` that “freeze language”.

## 4) Switching language

Switch language via the entry provided by the i18n service:

```ts
const i18n = yield* $.use(I18nTag)
yield* i18n.changeLanguage("zh")
```

## 5) Async readiness (ready / no-wait vs wait)

External i18n instances may require async initialization. `@logixjs/i18n` provides two render modes:

- `render(token, { fallback })`: no waiting; if i18n is not ready, it immediately returns a fallback (`fallback` or `key`).
- `renderReady(token, { fallback }, timeoutMs?)`: wait for readiness (default max 5s); on timeout/failure it returns a fallback.

This lets you choose between “fast availability” and “eventual consistency”. In most cases, prefer storing tokens in state and letting UI naturally render final text once ready. Only use `renderReady` when Logic must produce a final string.

## 6) Snapshot projection stays in app code

`@logixjs/i18n` now stays service-first. If you need `snapshot` inside module state or want to expose language switching as Actions, write a small projection helper in app code next to your Runtime boundary.

This keeps the package surface focused on `I18n.layer(driver)` and token contracts, and avoids carrying a second default authoring path inside the package.
