# STM（Tx*）在 Logix 的局部引入策略

## 1. 结论先行

- 引入时机：`logix-core` 通过 G1 后立刻做局部 PoC。
- 引入范围：仅 `WorkflowRuntime.ProgramState` 与 `ProcessRuntime` 控制面状态。
- 禁区：`ModuleRuntime.transaction`、`TaskRunner`、外部 IO step 执行体。
- 决策机制：MUST 全过 + SHOULD 至少 2 项通过，才可 `GO`。

## 2. 为什么不是“全迁完再做”

- 全迁后再做会导致外围包设计已固化，局部收益难以扩散。
- `G1 -> G2` 插入更利于在主干稳定后快速验证“是否值得”。

## 3. 为什么不是“一起全面做”

- 迁移风险与范式改造风险叠加，问题定位成本过高。
- 若 PoC 失败会拖慢全仓迁移节奏，违背主线优先原则。

## 4. 评估维度

- 正确性：乱序、重复提交、重复取消、回滚语义。
- 性能：p95、吞吐、内存（沿用 G1 红线）。
- 可诊断性：事件链完整性、定位步骤是否减少。
- 复杂度：目标模块分支/LOC/认知复杂度是否下降。

## 5. 失败处理

- 任何 MUST 失败 -> `NO-GO`。
- `NO-GO` 后回归非 STM v4 实现，不阻断 S4/S5/S6。
- 保留 PoC 证据与结论，作为后续独立 spec 的输入，而非悬挂债务。
