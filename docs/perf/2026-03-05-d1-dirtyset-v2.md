## 改了什么（实现点）

1. 事务脏证据统一：`TxnDirtyEvidence`
   - `packages/logix-core/src/internal/runtime/core/StateTransaction.ts`
   - 把「root-level dirty」与「list-level index hint」合并到一个事务窗口证据对象上，供下游统一消费。

2. validate 消费接口对齐
   - `packages/logix-core/src/internal/state-trait/validate.impl.ts`
   - `ValidateContext.txnIndexEvidence` -> `ValidateContext.txnDirtyEvidence`，list-scope / item-scope 的 `changedIndices` 统一从 evidence 读取。
   - 交易接线点：
     - `packages/logix-core/src/internal/runtime/core/ModuleRuntime.transaction.ts`

3. RowId reconcile 触发条件收敛（本刀主要收益点）
   - `packages/logix-core/src/internal/state-trait/rowid.ts`
   - 对「有 trackBy 的 list」：只有结构变化或 trackBy 字段脏才触发 `rowIdStore.updateAll`；行内字段更新不再触发 O(n) reconcile。
   - 对「无 trackBy 的 list」：保持旧行为（任何 list overlap 都触发），避免破坏 reference-identity 正确性。

4. 回归测试对齐
   - `packages/logix-core/test/internal/StateTrait/StateTrait.RefList.ChangedIndicesFromTxnEvidence.test.ts`
   - `packages/logix-core/test/internal/StateTrait/RowId.UpdateGate.test.ts`
   - `packages/logix-core/test/internal/Runtime/ModuleRuntime/ModuleRuntime.transaction.ListConfigsGuard.Perf.off.test.ts`

## 为什么有用（机制解释）

- 过去 list 的任意子字段更新都可能触发 rowId 全量 reconcile，导致「每次 row edit 都做一次 O(n)」的隐藏成本。
- D-1 把 reconcile gate 收敛到“结构变化/trackBy 变化”这两类必要条件；绝大多数 row edit（例如编辑某一行的业务字段）直接跳过全量扫。
- 同时把 txn 脏证据统一成 `TxnDirtyEvidence`，避免 validate/rowId/list-scope 各自重复解析 patch path/dirty roots。
