# 开发者主权 (Developer Sovereignty)

> "A platform should be a tool, not a jail."
> "Always Ejectable, Zero Lock-in."

在低代码（Low-Code）和模型驱动开发（MDD）的历史坟墓中，躺满了因为“强绑定”、“黑盒运行时”和“无法逃逸”而被开发者抛弃的产品。

Logix 深知：**要赢得开发者的信任，首先要尊重开发者的主权。**

## 1. 零锁定 (Zero Lock-in)

Logix 生成的代码**必须**是标准的、可读的、符合行业最佳实践的代码。

- **标准技术栈**：生成的是 TypeScript、React、Effect-TS，而不是某种 XML 配置或二进制大对象。
- **无黑盒运行时**：Logix Runtime (`@logixjs/core`) 是一个轻量的、开源的库，而不是一个庞大的闭源引擎。你完全可以阅读它的源码，甚至 Fork 它。

如果有一天 Logix 倒闭了，或者你不想续费了，你的项目**完全不受影响**。它就是一个标准的 TypeScript 项目，任何现代 IDE 都能打开和运行。

## 2. 随时逃逸 (Always Ejectable)

很多平台提供 "Eject" 按钮作为最后的手段（Nuclear Option），一旦点击就再也回不去。
Logix 提供的是 **"Clean Mode"** —— 一种温和的、随时可用的状态。

你可以选择：

- **Managed Mode**：享受 AI 的全自动化维护，Logic/Flow 保持同步。
- **Ejected Mode**：对某个文件进行人工魔改。平台会识别出“Hash 不匹配”，并自动退化为“辅助模式”——不再强行覆盖你的代码，而是通过锚点（Anchors）尝试注入更新，或者仅仅提供建议。

## 3. 双向尊重 (Mutual Respect)

传统的代码生成器往往是霸道的：`// DO NOT EDIT THIS FILE`。
Logix 提倡**双向尊重**的契约：

- **平台尊重开发者**：
  - 保留你的手动修改（通过 AST 级的 Diff/Merge）。
  - 允许你 Override 任意层级的默认行为。
- **开发者尊重平台**：
  - 遵循一定的目录结构规范（Convention）。
  - 不随意破坏关键的元数据锚点。

这种契约建立了一种“人机协作”的新常态，而不是“机器统治”或“机器单纯做苦力”。

## 4. 为什么这对 AI 编程很重要？

AI 目前的水平还达不到 100% 完美。开发者必须拥有 **"Human in the loop"** 的修正权。
如果 AI 生成的是一堆无法修改的黑盒代码，那么 AI 的哪怕 1% 的错误都会成为项目的致命阻塞点。

只有保证了“开发者主权”，保证了代码是“可读、可改、可逃逸”的，开发者才敢真正把核心业务交给 AI 去生成。
