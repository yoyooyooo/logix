# User Docs Writing Standard

## 目的

统一面向用户的文档写法、叙事密度、层级边界与验证门槛，保证 docs 站点长期保持单一口径、稳定默认路径和对 Agent 友好的 authoring surface。

## 与其他标准的关系

- `docs/standards/docs-governance.md` 负责 docs 路由、升格与回写协议
- 本文负责用户文档的写法、结构、术语和质量门
- `docs/ssot/**`、`docs/adr/**`、`docs/standards/**` 继续分别承担事实、裁决与跨主题规范

## 单一权威

- 用户文档的事实依据优先来自 `docs/ssot/**`
- 重大方向与边界变化优先来自 `docs/adr/**`
- 跨主题写作规范与统一护栏固定在 `docs/standards/**`
- `apps/docs/content/docs/**` 是对外用户文档承载面，不自行演化第二套事实源

## 总原则

### 先对齐事实，再组织叙事

- 先核对 `docs/ssot/**`、`docs/adr/**`、`docs/standards/**`
- 再核对源码当前行为与公开 API
- 文档可以领先代码，但不能脱离当前事实源
- 若源码与事实源冲突，优先回到事实源统一口径，再决定实现如何收敛

### 默认路径优先

- 每个主题先给一条最小 canonical route
- 默认路径应足够覆盖多数用户场景
- 高级路径只在存在明确触发条件时展开
- 若存在多条可行路径，先写最小、最稳、最一致的一条

### 明确能力边界

- 每页都应说明能力属于哪一层，例如 `core`、`react`、`form`、`toolkit` 或应用侧自定义封装
- 每页都应说明什么属于默认公开面，什么属于进阶路径或应用侧责任
- 不把 example、内部 helper 或业务临时封装直接写成官方默认路径

### 面向开发者，也面向 Agent

- 标题、段落和代码示例应能直接被提炼为稳定 authoring pattern
- 减少含糊语句、双重默认、模糊条件和竞争性表达
- 一页内尽量只保留一条主链心智，避免让读者在多个看似等价的入口之间自行裁决

## 页面结构

### 开头

每页开头至少要回答四个问题：

1. 这是什么
2. 解决什么问题
3. 默认怎么用
4. 边界在哪里

推荐顺序：

- 一段总述，说明对象与默认定位
- 若有必要，紧跟一个非常短的默认路径说明
- 避免写作者视角、自我说明式引子或课堂式导览

### 正文

正文优先按以下结构组织：

1. 角色或对象
2. 核心语义
3. 默认用法或最小示例
4. 约束、边界与常见误用
5. 需要时再进入高级路径、控制面或特例

### 收尾

- 页尾统一使用 `See also / 延伸阅读`
- 只链接真正补充语义或下一步下钻所需的页面
- 不写“恭喜你完成了”这类教学收尾

## 叙事风格

### 应该写成什么样

- 客观、直接、手册式
- 信息密度高，但句子保持可扫描
- 用概念、语义、约束和默认路径组织内容
- 让同一主题在中英文页上保持相同结构重心

### 不应写成什么样

- 不写课件腔，例如“这一页做什么”“你会学到什么”“读完你将获得”
- 不写对话腔、自我说明腔或过程式旁白
- 不写鼓励式、庆祝式、教学脚本式收尾
- 不把维护者内部心智直接投射成用户文档

## 术语与命名

### 固定术语

- 同一概念固定一个名字
- 同一名字固定一个含义
- 新术语引入后，相关页面同步回写，避免并行口径

### 术语收口

优先使用稳定、可长期维护的词：

- `default route`
- `advanced route`
- `fixed-root semantics`
- `service-first`
- `canonical route`

默认避免使用容易带入阶段性或内部味的词：

- `blueprint`
- `day-one route`
- `expert route`
- 任何未在事实源中稳定收口的维护者内部简称

## 代码示例

- 示例优先使用最小可运行切片
- 示例名称保持稳定，避免同一主题频繁换名
- 示例应直接体现推荐 authoring pattern
- 示例代码与当前公开 API 保持同口径
- 若存在更短、更稳的写法，优先保留那一条
- 复杂示例要分段说明状态结构、动作、并发、错误处理和边界

## 各目录的职责

### `get-started`

- 提供最短闭环
- 目标是让用户尽快跑通一个小切片
- 避免提前灌入高级控制面或大量分支

### `essentials`

- 承担主链心智
- 说明核心不变量、默认语义和基本组合方式

### `learn`

- 拆解概念边界和语义差异
- 讲清多个对象之间如何分工
- 不把高阶控制面混进基础心智页

### `patterns`

- 沉淀稳定可复用模式
- 重点说明状态结构、动作、并发、回滚、联动和边界

### `recipes`

- 面向迁移、集成与具体场景落地
- 重点说明输入条件、推荐组合和落地结果

### `advanced`

- 承担控制面、调度、诊断、性能和回退
- 前提是默认路径已经在其他目录交代清楚

### `api`

- 作为契约页
- 重点说明公开面、参数、返回值、语义、限制和相关页面

### `form`

- 作为领域专题
- 说明 form 自身默认路径、边界以及何时下沉到 `core` 或 `react`

## 中英文页一致性

- 中英文页要共享同一结构和语义分层
- 允许自然表达，不要求逐句直译
- 不能出现中文页是教程腔、英文页是参考手册，或反过来的分叉

## 事实对齐规则

- Example 只能作为材料来源，不能天然升格为官方推荐写法
- 业务项目里的自定义 helper、sugar、support，只有在边界稳定后才进入官方文档
- 若用户文档给出推荐路径，必须能在当前事实源或当前实现中找到明确依据

## 质量门

### 回写范围

涉及用户文档统一口径时，默认至少同时核对：

- 对应中英文页
- 所在目录的 `meta.json` / `meta.cn.json`
- 相关索引页
- 相邻页面的 `See also / 延伸阅读`

### 机械校验

- `git diff --check`
- 批量改写后的关键词扫描，清理教程腔和旧术语残留
- 站点相关改动至少运行：
  - `pnpm -C apps/docs types:check`
  - `pnpm -C apps/docs build`

### 构建阻塞

- frontmatter、MDX、链接与目录元数据错误要在当前任务内修完
- 对本次改动无关但会阻塞 docs build 的问题，也应一并修复

## 与 Agent 协作相关的补充要求

- 每页都要明确默认路径，减少 Agent 自行选路空间
- 每页都要明确能力归属层级，减少跨层误用
- 每页都要明确何时需要下钻到 advanced route
- 示例要能被 Agent 直接复用或改写，避免依赖隐含上下文

## 一句话结论

用户文档应以事实源为基准，围绕默认路径、清晰边界和稳定术语组织叙事，并通过中英文对齐与站点级验证保持长期一致性。
