# S2 Diagnostics vs S0 Snapshot

更新时间：2026-03-07

## 目的

对照 `diagnostics/s0.snapshot.md`，确认 Stage 2 当前实现没有降低诊断可解释性，至少满足：
- 事务路径仍能给出稳定锚点与可序列化证据；
- lifecycle / workflow 路径的诊断事件仍可序列化、可回放；
- `off/light/full` 三档诊断语义仍然成立；
- 关键字段（`instanceId/txnSeq/opSeq/runtimeLabel`）仍保持可读、可审计。

## 对照结果

### 1. 事务路径

- `TxnLaneEvidence.Schema` 继续通过：说明 `txnLane` 证据包仍满足 schema 约束，`txnSeq`/anchor 等关键字段没有漂移。
- `RuntimeEvidencePackage` 继续通过：说明 runtime 级观测包仍可序列化，并保留服务摘要与实例标识。

### 2. 工作流 / 生命周期路径

- `Lifecycle.DiagnosticsSerialization` 继续通过：说明 init/start/destroy 三类 lifecycle 事件仍能稳定序列化。
- 当前没有证据表明 `WorkflowRuntime` 迁移引入新的 diagnostics 字段断裂；本轮回归重点覆盖的是 shared serialization contract，而不是 workflow 行为性能。

### 3. 错误 / 诊断分级路径

- `Debug.DiagnosticsLevels` 继续通过：`off/light/full` 三档语义仍成立。
- `Debug.RuntimeDebugEventRef.Serialization` 继续通过：Debug 事件转 RuntimeDebugEventRef 的可序列化 contract 未退化。

## 结论

- 相对 `s0.snapshot.md` 的目标，当前 Stage 2 诊断链路没有出现“可解释性下降”的证据。
- `T031` 可以视为完成：S0 里要求补齐的 slim 可序列化 / diagnostics levels / 核心锚点对照，当前已有直接回归测试覆盖。
- 后续若继续推进 `T020/T021/T022`，需要保持这组测试持续为绿；任何一项变红都应视为 diagnostics regression，而不是普通重构噪声。
